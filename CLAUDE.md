## Architecture

**KranzTV** is a retro cable TV experience. Videos play on a deterministic, stateless schedule — no database, no server sync. The core constraint is that every viewer of a channel sees the same video at the same moment.

### Scheduling (the central abstraction)

`src/lib/scheduling/algorithm.ts` — `getSchedulePosition(channel, timestamp)` is a **pure function**: given a channel and a timestamp, it deterministically returns which video is playing and at what seek position. It runs identically on client and server.

The algorithm:

1. `secSinceMidnight` = seconds since UTC midnight
2. `dayOffset` = `(daysSinceEpoch × 127) % totalDurationSeconds` — daily rotation using prime number for distribution
3. `cyclePos` = `(secSinceMidnight + dayOffset) % totalDurationSeconds`
4. Walk `channel.videos` until accumulated duration exceeds `cyclePos`

**Never add server state or randomness to scheduling logic.**

### Data flow

```
ChannelPreset (presets.ts) or imported Channel (localStorage)
  → buildChannel(preset, apiKey)          # fetches YouTube playlist + video details
  → Channel (videos[], totalDurationSeconds)
  → getSchedulePosition(channel, now)     # pure, runs on 1s tick
  → SchedulePosition (video, seekSeconds, slotStart, slotEnd)
  → TvPlayer (YouTube IFrame API)
```

YouTube Data API calls happen **client-side** via `VITE_YOUTUBE_API_KEY`. If no key is present, `buildMockChannel()` is used — the app stays fully functional with 3 hardcoded videos.

### Channel import

`src/lib/import/` contains the full import pipeline:

- `parser.ts` — `extractPlaylistId(input)`: pure function, extracts a YouTube playlist ID from any URL format or bare ID
- `schema.ts` — `ImportFormSchema` (Zod), `channelToPreset()`, `getNextChannelNumber()`
- `import-channel.ts` — `importChannel(url, name, nextNumber, apiKey)`: fetches playlist + video details, returns `ImportResult`

Custom channels are persisted to localStorage via `src/lib/storage/local-channels.ts` and hydrated into `TvLayoutContext` on mount. They appear in the guide grid and are navigable via ↑↓ alongside preset channels.

Key component directories:
- `src/components/epg-overlay/` — full-screen or inline EPG grid (header, time axis, rows, cells)
- `src/components/info-panel/` — desktop right-panel showing current channel/video info and mini channel list
- `src/components/import-wizard/` — import modal for adding custom channels via playlist URL

### Path aliases

All three aliases (`#/*`, `@/*`, `~/`) resolve to `./src/`. Use `~/` by convention in source files.

### Routing (TanStack Start)

- `_tv.tsx` — layout route: three-panel desktop (video 2/3 + info panel 1/3 + EPG bottom) or full-width (tablet/fullscreen); uses React context (`useTvLayout`) to share channel state, guide/import visibility, and custom channels with child routes
- `_tv.channel.$channelId.tsx` — channel view; loads channel data (preset or custom), manages player state
- `api/channels.ts` — server function via `createServerFn` (not a REST endpoint file — call `getChannels()` directly from client code)
- `routeTree.gen.ts` — auto-generated by TanStack Router; **do not edit manually**

### Key hooks

- `use-current-program.ts` — polls `getSchedulePosition` every second, returns live `SchedulePosition`
- `use-channel-navigation.ts` — prev/next channel via router; accepts `allChannels` array so custom imported channels are included in navigation
- `use-keyboard-controls.ts` — global keyboard bindings (↑↓ channels, G guide, M mute, N now-playing, H home, F fullscreen, V overlay, I import, ? help, Esc close)
- `use-is-desktop.ts` — media query hook for 1024px+ breakpoint (used by layout to switch between three-panel and full-width)

### YouTube IFrame API

`src/lib/player/youtube-iframe.ts` wraps the YouTube IFrame Player API. The player is initialized once and reused across channel switches — `loadVideoById` is called with `seekSeconds` on each new position.

### Observability

- Server APM: `dd-trace` initialized via `--require dd-trace/init` in Docker CMD only (not in dev)
- Browser: `src/lib/datadog/rum.ts` tracks custom actions (`channel_switch`, `guide_toggle`, `import_started`, `keyboard_shortcut`)
- Custom metrics via DogStatsD: `kranz_tv.channel.switch`, `kranz_tv.video.playback`

## Environment Variables

| Variable                   | Required           | Description                                                            |
| -------------------------- | ------------------ | ---------------------------------------------------------------------- |
| `VITE_YOUTUBE_API_KEY`     | No (mock fallback) | YouTube Data API v3 key                                                |
| `DD_API_KEY`               | Docker only        | Datadog APM                                                            |
| `VITE_DD_RUM_APP_ID`       | No                 | Browser RUM                                                            |
| `VITE_DD_RUM_CLIENT_TOKEN` | No                 | Browser RUM                                                            |
| `VITE_DD_COMMIT_SHA`       | No (CI sets it)    | Git SHA baked into RUM global context for traceability; set by CI only |
| `DD_VERSION`               | Docker only        | Server APM version; set at Docker deploy time to match `package.json`  |

App version (`__APP_VERSION__`) is read from `package.json` at Vite build time — no env var needed. It is automatically correct for all build environments (CI, Cloudflare, local).

## Testing

Tests live in `tests/unit/`, `tests/integration/`, `tests/e2e/` and `src/test/` (setup file). The scheduler (`algorithm.ts`, `time-utils.ts`, `epg-builder.ts`) is pure and fully unit-testable — new scheduling logic must have unit tests. Test setup is in `src/test/setup.ts`.

## TypeScript

Strict mode with `noUnusedLocals`, `noUnusedParameters`, `noFallthroughCasesInSwitch`. All YouTube API responses are validated with Zod schemas before use (`src/lib/channels/youtube-api.ts`).
